# 代码审查:v1.0.5 变更集

## 概述

v1.0.5 版本使用 NeoForge 原生的 `ModConfigSpec` API 替换了 Cloth Config,并为 `csgo_key3` 新增了一个锻造配方。迁移在结构上是合理的 —— Cloth Config 依赖已移除、import 已更新、新的锻造 JSON 包含所有必需字段 —— 但新的 `CsboxConfig` 类存在一个**静默的初始化 bug**,会在运行时破坏所有由配置驱动的行为:没有音量、没有生物掉落、没有调试日志、默认 box 永不加载,而动画速度枚举的解引用会在 `CsboxProgressScreen` 第一次读取它时抛出 NPE。此外,新的锻造配方(以及先前就存在的钥匙配方)被提交到了遗留的复数 `recipes/` 目录,而不是规范的 `recipe/` 数据包路径,因此它们无法在游戏中加载。v1.0.4 中服务端权威的 RNG 机制得以保留(数据包代码未改动)。

## 审查发现

### CR-001:`CsboxConfig.init()` 方法已定义但从未被调用 —— 运行时所有配置驱动的值都读取到 0/null

- **严重度**:严重
- **文件**:`src/main/java/com/reclizer/csgobox/config/CsboxConfig.java:76-87`(以及所有消费位置)
- **描述**:`CsboxConfig` 采用了 "init()" 模式:构造函数存储 `*Value` 引用(`ModConfigSpec.ConfigValue<T>`),而 `init()` 预期在配置加载时被调用,以填充普通的 Java 字段(例如 `this.openSoundVolume = this.openSoundVolumeValue.get()`)。然而 `init()` 在整个代码库中**从未被调用** —— 没有任何 `ModConfigEvent.Loading` / `ModConfigEvent.Reloading` 处理器订阅(grep 已确认:只存在构造函数的 `registerConfig` 调用)。由于 Java 基本类型默认为 0/false,且 `AnimationSpeed` 枚举引用默认为 null,所有消费位置都会读取到错误的值:

  | 字段 | 读取位置 | 运行时实际值 | 用户可见的影响 |
  |---|---|---|---|
  | `globalDropRatePercent` | `ModEvents.java:41` | 0 | **生物掉落 CS:GO box:永不发生**(`0/100F * rate = 0` → 随机检查总是失败) |
  | `openSoundVolume` | `ClickEvent.java:37` | 0 | 打开音效永远不会播放 |
  | `tickSoundVolume` | `CsboxProgressScreen.java:215` | 0 | 动画过程中的 tick/ding 音效永远不会播放 |
  | `finishSoundVolume` | `CsLookItemScreen.java:33` | 0 | 揭晓音效永远不会播放 |
  | `loadDefaultBoxes` | `CsgoBox.java:70` | false | 默认 box 永不自动加载(配置 schema 默认为 `true`,但 Java 字段保持 `false`) |
  | `enableDebugLogging` | `CsgoBox.java:77` | false | 调试日志永久关闭 |
  | `showItemNames` | `CsboxScreen.java:229` | false | 物品名称在 box 预览中隐藏(schema 默认 `true`) |
  | `animationSpeed` | `CsboxProgressScreen.java:249` | **null** | **抛出 `NullPointerException`** —— 动画屏幕的第一次 tick 就会崩溃 |

  项目自身的 `AGENTS.md`(第 21 行)记录了预期的模式:`builder.defineInRange("openSoundVolume", 100, 0, 100).get()` —— 即在构造函数中立即调用 `.get()`(无 `init()` 间接层)。当前的实现两者都没做。
- **建议修复**:方案(a)按 `AGENTS.md` 的规定在构造时内联 `.get()` —— 将每个 `this.XValue = builder.define...(...)` 替换为 `this.X = this.XValue = builder.define...(...).get()` 并删除 `init()`;或方案(b)将 `init()` 连接到真实事件,在 `CsgoBox.java` 中添加:

  ```java
  @SubscribeEvent
  public static void onConfigLoad(ModConfigEvent.Loading event) {
      if (event.getConfig().getSpec() == CONFIG_SPEC) {
          CONFIG.init();
      }
  }
  @SubscribeEvent
  public static void onConfigReload(ModConfigEvent.Reloading event) {
      if (event.getConfig().getSpec() == CONFIG_SPEC) {
          CONFIG.init();
      }
  }
  ```

  方案(a)更简单,且与文档化约定一致。

### CR-002:`switch (CONFIG.animationSpeed)` 在第一个动画 tick 时会抛出 NullPointerException

- **严重度**:严重
- **文件**:`src/main/java/com/reclizer/csgobox/gui/CsboxProgressScreen.java:249`
- **描述**:`CsgoBox.CONFIG.animationSpeed` 是在 `CsboxConfig.java:18` 声明的普通 Java 枚举字段。因为 `init()` 从未被调用(CR-001),该字段保持 Java 默认值 `null`。对 null 枚举引用进行 `switch` 语句会抛出 `NullPointerException`(Java SE 17+ 对引用类型 `switch` 的行为)。这会在任何客户端打开 CS:GO box 的瞬间崩溃 —— 来自 `CsboxProgressScreen` 构造函数对 `readAnimationTicks()` 的第一次调用就会抛出,根本无法构造动画屏幕(调用点:`this.totalTicks = readAnimationTicks();`,第 60 行)。
- **建议修复**:修复 CR-001 即可覆盖。`init()` 运行后(或字段在构造时被填充后),该值默认为 `AnimationSpeed.NORMAL`。可选地在调用点添加 null 守卫以纵深防御。

### CR-003:`globalDropRatePercent` 读取值始终为 0 —— 生物掉落 CS:GO box 的功能完全被禁用

- **严重度**:严重
- **文件**:`src/main/java/com/reclizer/csgobox/event/ModEvents.java:41`
- **描述**:这是 CR-001 应用到服务端权威游戏系统上的用户可见后果。因为 `CONFIG.globalDropRatePercent` 始终为 0,`effectiveRate *= CsgoBox.CONFIG.globalDropRatePercent / 100F` 这一行将每个实体的掉落率都归零。下一行 `if (effectiveRate > 0 && RANDOM.nextFloat() < effectiveRate)` 会在 `0 > 0` 处短路并跳过 `spawnAtLocation` 调用。预期行为(默认 100% 倍率,根据 `CsboxConfig.java:39` 可配置为 0–1000%)在 v1.0.5 中完全失效 —— 玩家无法通过击杀生物获得 box,无论在 `config/csgobox-common.toml` 中写入什么值都无效。

  v1.0.4 在这一行有 `CONFIG != null` 守卫,掩盖了 v1.0.4 中的"配置缺失"bug(因为 `CONFIG` 实际上由于一次损坏的 `b7b11e5` 重定位而未被初始化 —— 见 CR-004)。v1.0.5 的 diff 正确地移除了该守卫,理由是 `CONFIG` 是 `final`。这个推理是对的,但底层的字段初始化是有问题的。
- **建议修复**:修复 CR-001。在 `CONFIG.globalDropRatePercent` 真正接收到其 `defineInRange` 值后,此代码将按设计工作。

### CR-004:新的锻造配方被提交到错误的数据包目录 —— 游戏无法加载它

- **严重度**:严重
- **文件**:`src/main/resources/data/csgobox/recipes/csgo_key3_smithing.json:1-16`
- **描述**:现代 Minecraft(1.21+,包括项目在 `gradle.properties` 中的 `minecraft_version_range=[1.21.1,1.22)`)从 `data/<namespace>/recipe/`(单数)扫描配方数据包。新的 `csgo_key3_smithing.json` 被添加到了 `src/main/resources/data/csgobox/recipes/`(复数)下。`RecipeManager` 不会发现 `recipes/` 中的文件,因此 `csgo_key3` 的锻造配方在游戏中会被静默地不注册 —— 玩家在生存模式下没有任何方式可以合成下界合金钥匙。

  这个 bug 也影响到从 v1.0.4 沿用下来的旧钥匙配方(`csgo_key0.json`、`csgo_key1.json`、`csgo_key2.json`)—— 它们也在相同的错误路径上。项目自身的 `AGENTS.md`(第 58 行)明确记录了正确路径:`src/main/resources/data/csgobox/recipe/`("注意是单数 `recipe`")。此次发布违反了文档化约定。

  注意:HEAD 工作树中的文件已经在 `recipe/`(单数)下 —— 这次重命名发生在发布后的 "docs" 提交 `24a82ac` 中,不属于 v1.0.5 tag 的一部分。按 tag 标记的 v1.0.5 发布随附的是损坏的路径。
- **建议修复**:将 `src/main/resources/data/csgobox/recipes/csgo_key3_smithing.json`(以及 `csgo_key0.json`、`csgo_key1.json`、`csgo_key2.json`)上移一层到 `src/main/resources/data/csgobox/recipe/`。使用 `git mv` 并将重命名纳入发布提交。构建后验证生成的 jar 包含 `data/csgobox/recipe/csgo_key3_smithing.json`。

### WR-001:v1.0.4 中既有的 `CONFIG != null` 守卫已经表明配置不可靠

- **严重度**:警告
- **文件**:`src/main/java/com/reclizer/csgobox/event/ModEvents.java:41`(v1.0.5 之后的形式)
- **描述**:v1.0.4 的 `ModEvents.java` 有 `if (CsgoBox.CONFIG != null) { effectiveRate *= ... }`,因为在运行时 `CONFIG` 实际上可能为 null(v1.0.4 源码树中根本没有 `CsboxConfig.java` 文件 —— 它是在 v1.0.5 周期内的 `b7b11e5` 提交中才引入的)。v1.0.5 的 diff 正确地移除了该守卫,理由是 `CONFIG` 现在是 `static final`。然而,按 tag 标记的 v1.0.4 发布在源码层面已经损坏:`import com.reclizer.csgobox.config.CsboxConfig;` 解析到了一个不存在的类。任何从 `a86c8c4`(v1.0.4 tag)构建的消费者都会编译失败,或者如果从其他源码状态编译出来,会遇到 `CONFIG == null` 而永远不掉落 box。v1.0.5 是第一个真正能干净编译的版本 —— 但根据 CR-001/CR-003 它仍然存在初始化 bug。
- **建议修复**:v1.0.5 本身无需修复 —— 守卫移除是正确的。在发布检查清单中增加一项:在 `config/csgobox-common.toml` 中设置 `globalDropRatePercent = 200`,启动一个有已知掉落实体的世界,确认掉落率翻倍。

### WR-002:ModConfigSpec builder 的 `defineListAllowed`/`worldRestart` 语义值得文档化

- **严重度**:警告
- **文件**:`src/main/java/com/reclizer/csgobox/config/CsboxConfig.java:32-74`
- **描述**:schema 正确地使用 `defineInRange`/`define`/`defineEnum` 定义(最小/最大范围正确,默认值与 schema 注释一致),builder 正确使用 `push("general")`/`push("advanced")`/`pop()` 来嵌套 TOML 分区。两个小的健壮性缺口:

  1. schema 注释记录了分类,而扁平的 Java 端字段不再反映这些分类(例如注释说"全局掉落率乘数,以百分比表示(0-1000,默认 100)",在 TOML 中位于 `[general]` 下)。这在功能上没问题,但意味着面向用户的 TOML 分区注释不再与 Java 字段名匹配 —— 阅读文件的玩家可能会感到困惑。`AGENTS.md:38` 记录了这个约定,所以这是有意的,不是回归。
  2. 没有任何值使用 `.worldRestart()` 或 `.gameRestart()` 标记。如果玩家在游戏中修改 `loadDefaultBoxes`,`commonSetup` 时的 `BoxJsonLoader::loadAll` 不会重新运行 —— 但这也是 v1.0.4 的行为,不是回归。

  这些都不是 bug,而是未来加固时的质量观察。
- **建议修复**:可选:为 `loadDefaultBoxes` 添加 `.worldRestart()`,这样实时修改它会通过 `/reload` 或重启真正触发重载。其他值(音量、掉落率、调试日志)热重载是没问题的。

### IF-001:`CsboxConfig` 类名仍然使用 "Csbox" 前缀,与 `csgobox` 命名空间不一致

- **严重度**:信息
- **文件**:`src/main/java/com/reclizer/csgobox/config/CsboxConfig.java:5`
- **描述**:该类名为 `CsboxConfig`,但位于 `csgobox` 包下,引用形式为 `CsgoBox.CONFIG`/`CsboxConfig`。用户提供的提示中提到这是"正在进行的、有意的机械重命名,不是 bug" —— 仅为完整性标记,以便未来的重命名(例如 `CsgoboxConfig` 以匹配包名)可被跟踪。该类通过 `CsgoBox.CONFIG`(类型为 `CsboxConfig`)对外公开,因此重命名对任何导入 `com.reclizer.csgobox.config.CsboxConfig` 的外部 addon 来说都是破坏性 API 变更。
- **建议修复**:v1.0.5 暂不处理。如果要重命名,将其置于主版本号升级之后,并更新 `AGENTS.md`、`gradle.properties` 以及所有 `grep` 到的引用。

## 跨切面说明

- **v1.0.4 的服务端权威开启机制得以保留。** `PacketCsgoProgress` / `PacketBoxOpenResult` 在 v1.0.5 的 diff 中未变化,`CsboxProgressScreen` 仍然从服务端的 `PacketBoxOpenResult`(第 146–166 行)独占地获取 `serverWinningIndex`、`resultItem`、`resultGrade`、`animationItems`、`animationGrades`。最终奖励的 RNG 不在客户端执行。
- **Cloth Config 已完全移除。** `build.gradle` 不再声明 `me.shedaniel.cloth:cloth-config-neoforge`;`neoforge.mods.toml` 中的 `[[dependencies.csgobox]] modId="cloth_config"` 块已删除;`src/` 中没有 `me.shedaniel.autoconfig` / `me.shedaniel.cloth` 的 import。对 `ModConfigSpec` 的迁移在结构上是正确的 —— 唯一缺陷是 `init()` 步骤未被连接。
- **配方 JSON 有效性。** 新的 `csgo_key3_smithing.json` 包含所有四个必需字段(`type`、`template`、`base`、`addition`、`result`),格式正确;`type: minecraft:smithing_transform` 是 MC 1.21.1 在 1.20+ 中正确的配方类型。唯一缺陷是目录路径(CR-004)。
